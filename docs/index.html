1. Executive Summary

This is a healthcare benefits correspondence automation system built on the Pick/MultiValue (UniData) platform. It manages three key categories of regulatory and benefits notices:

  • COBRA notices — Initial introduction letters and dependent notices for COBRA continuation coverage
  • HIPAA privacy notices — Notice of Privacy Practices for both self-funded (SF) and fully-insured (FI) groups
  • Disabled dependent letters — Review requests, recertifications, approvals, denials, and no-response letters for disabled dependents

The system is implemented through 3 main programs:

ProgramPurposeOutput
COBRANProcess COBRA initial noticesSmartComm correspondence for employees and dependents
HIPAA.NPPGenerate HIPAA Notice of Privacy PracticesSmartComm (SF) or PCL print files via FTP (FI)
PRINT.DISABLED.DEP.LETTERSGenerate disabled dependent letters (6 types)SmartComm correspondence

These programs interact with shared data records including EMPLOYEES, EMPLOYERS, EROPTIONS, DEPENDENTS, and ERADJ, and submit correspondence through a centralized *CREATE.CORRESP.TRACKING subroutine that interfaces with the SmartComm document management system.

Key Finding: Significant functional coupling exists across the 3 programs, with 8+ shared concerns (eligibility filtering, employer loading, billing, audit) duplicated in each. This report provides a detailed decoupling strategy with 8 proposed extracted subroutines.

7. Decoupling Strategy

The following 8 extracted subroutines are proposed to decouple the shared concerns:

7.1 *CHECK.LETTER.ELIGIBILITY

Proposed Signature:

SUBROUTINE *CHECK.LETTER.ELIGIBILITY(IS.ELIGIBLE, EEID, MAT EEREC, MAT ERREC, MAT EOREC, LETTER.TYPE, TODAY)

Extracts from:

  • COBRAN — MAIN.LOOP (lines 196–271): status checks, EENEED='X' skip, LOCATE effective date, termination/COBRAFEE checks
  • HIPAA.NPP — FILTER.EEIDS (lines 458–556): status exclusions, coverage checks, HIPAA.SENT cutoff comparison
  • PDDL — VALIDATE.DEP.DATA + CHECK.DEP.NEED (lines 303–467): DEP.NEED validation, same-day duplicate check

Current Duplicated Pattern:

* Pattern repeated in all 3 programs:
IF INDEX(EENEED,'X',1) > 0 THEN CONTINUE
LOCATE TODAY IN EEEFFDATE<1,1> BY 'DR' SETTING EEIDX ELSE NULL
IF EEEFFDATE<1,EEIDX> = '' AND EEIDX > 1 THEN EEIDX -= 1
* ... then program-specific status/coverage checks

Extraction: Consolidates the common effective-date LOCATE, EENEED='X' exclusion, and status filtering into a single subroutine. Each program passes its LETTER.TYPE ('COBRA', 'HIPAA', 'DDL') to invoke the appropriate eligibility rules. Returns IS.ELIGIBLE flag and sets EEIDX.

7.2 *LOAD.EMPLOYER.CONTEXT

Proposed Signature:

SUBROUTINE *LOAD.EMPLOYER.CONTEXT(LOAD.OK, FILENO, MAT ERREC, MAT EOREC, LAST.FILENO)

Extracts from:

  • COBRAN — MAIN.LOOP (lines 232–260): conditional read when FILENO changes
  • HIPAA.NPP — SET.SKIP.FILENO (lines 743–775): reads EMPLOYERS + EROPTIONS with error handling
  • PDDL — GENERATE.DD.LETTERS (lines 691–705): reads EMPLOYERS per dependent

Current Duplicated Pattern:

IF EEFILENO<1,EEIDX> <> FILENO THEN
   FILENO = EEFILENO<1,EEIDX>
   CALL *READREC('',MAT ERREC,EMPLOYERS,FILENO,'')
   IF @USER.RETURN.CODE # '' THEN ... CONTINUE
   CALL *READREC('',MAT EOREC,EROPTIONS,FILENO,'')
   IF @USER.RETURN.CODE # '' THEN ... CONTINUE
END

Extraction: Encapsulates the cached employer+eroptions read pattern. Compares FILENO against LAST.FILENO; if different, reads both records and returns LOAD.OK. Eliminates 3 copies of the same caching logic.

7.3 *SUBMIT.LETTER.REQUEST

Proposed Signature:

SUBROUTINE *SUBMIT.LETTER.REQUEST(ERROR.MESSAGES, FILENO, RECIPIENT.ID, DOCUMENT.NAME, PROGRAM.NAME, ADDITIONAL.DEPNO, ADDITIONAL.PURPOSE)

Extracts from:

  • COBRAN — SUBMIT.CORRESP.REQUEST (lines 445–594): EE and DP letter paths
  • HIPAA.NPP — SUBMIT.CORRESP.REQUEST: SF SmartComm submission
  • PDDL — SUBMIT.CORRESP.REQUEST (lines 766–821): per letter-type submission

Current Duplicated Pattern:

ERROR.MESSAGES = ''
CORRESP.DATA.REC = ''
CORRESP.DATA.REC<2> = FILENO
CORRESP.INPUT.DEF.ID = 'BASIC.SMARTCOMM.DOC.1'
RECIPIENT.TYPE = 'DEPENDENT'
METHOD.REQUESTED = 'LOOKUP'
REQUEST.SOURCE = PROGRAM.NAME
CALL *CREATE.CORRESP.TRACKING(ERROR.MESSAGES,...)

Extraction: Wraps the CORRESP.DATA.REC assembly and *CREATE.CORRESP.TRACKING call. All 3 programs use identical parameter setup; only FILENO, RECIPIENT.ID, DOCUMENT.NAME, and the optional additional-depno fields vary.

7.4 *ADD.NOTICE.BILLING

Proposed Signature:

SUBROUTINE *ADD.NOTICE.BILLING(EEID, FILENO, MAT EEREC, MAT EAREC, MAT EOREC, MAT ERREC, DIST.TYPE, EEIDX, EOIDX)

Extracts from:

  • COBRAN — DO.BILLING (lines 598–684): COBRACHG distribution, fee from EOCOBRA.NOTICE
  • HIPAA.NPP — DO.BILLING: HIPAA distribution, fee from EOHIPAAP

Current Duplicated Pattern:

* Both programs: locate ERADJ record, increment-or-create
BILLPERIODO = CONVERT('-','',OCONV(ERBILLEDTO+1,'D2-YM'))
SUFFIX = ''
LOOP
   FILENO.KEY = FILENO:BILLPERIODO:SUFFIX
   SUFFIX += 1
   CALL *READREC('',MAT EAREC,ERADJ,FILENO.KEY,'WAIT')
   * ... find or create matching ERADJ record
REPEAT
* Update EADISTFIELD, EADISTAMOUNT, EACOVPERIOD, EAAMOUNT
CALL *WRITEREC('',MAT EAREC,ERADJ,FILENO.KEY,'')

Extraction: Parameterizes the distribution type (COBRACHG vs HIPAA) and the fee source field. The increment-or-create ERADJ loop is identical in both programs.

7.5 *GENERATE.FI.HIPAA.NOTICE

Proposed Signature:

SUBROUTINE *GENERATE.FI.HIPAA.NOTICE(PCL.OUTPUT, EEID, MAT EEREC, MAT ERREC, FILENO, FORMS.DATA)

Extracts from:

  • HIPAA.NPP — GET.HIPAA.FORMS + GENERATE.HIPAA.FORMS: CERTS.LAYOUT lookup, PCL generation for fully-insured groups

Current Pattern: The FI path in HIPAA.NPP reads CERTS.LAYOUT by insurance carrier, merges fields from CERTS.MERGE, applies text from CERTS.TEXT2, and generates PCL output with embedded form data. This is a self-contained concern that only exists in HIPAA.NPP today.

Extraction: Isolates the FI-specific PCL generation into a reusable subroutine. While currently only called by HIPAA.NPP, extracting it simplifies the main program and enables future reuse if FI notices are needed elsewhere.

7.6 *SUBMIT.PRINT.FTP

Proposed Signature:

SUBROUTINE *SUBMIT.PRINT.FTP(FTP.STATUS, FILENAME, FTP.ID, PROGRAM.NAME)

Extracts from:

  • HIPAA.NPP — FTP.FILE subroutine: copies PCL to REDCARD.PCL, submits FTP request with FTP.ID 103069

Current Pattern: The FTP transmission creates an FTP.REQUESTS record, writes to FTP.LOG, and uses binary transfer mode for PCL files. Currently tightly coupled to HIPAA.NPP's filename generation logic.

Extraction: Parameterizes the filename, FTP.ID, and program name. Makes the FTP submission reusable for any future print-file-based correspondence delivery.

SUBROUTINE *UPDATE.DEP.LETTER.STATUS(MAT DPREC, EEID.DEPNO, CUR.NEED, ERROR.DEP.NEED, TODAY, PROGRAM.NAME)

Extracts from:

  • PDDL — WRITE.DEP.REC (lines 883–906) + UPDATE.DEP.NEED (lines 859–881): updates DPDDL.SENT, DPDDL.LETTERS, clears/retains DPDEP.NEED based on errors

Current Pattern:

* UPDATE.DEP.NEED: track which letters were sent today
IF DPDDL.SENT<1,1> = TODAY THEN
   * append to existing date's letters
   TEMP.DDL.LETTERS = CONVERT(',',@FM,DPDDL.LETTERS<1,1>)
   LOCATE CUR.NEED IN TEMP.DDL.LETTERS<1> SETTING XXX ELSE
      INS CUR.NEED BEFORE TEMP.DDL.LETTERS<-1>
   END
END ELSE
   * insert new date entry
   INS TODAY BEFORE DPDDL.SENT<1,1>
   INS CUR.NEED BEFORE DPDDL.LETTERS<1,1>
END

* WRITE.DEP.REC: clear or retain DPDEP.NEED based on errors
IF MAX.ERR.NEED = 0 THEN
   DPDEP.NEED = ''
END ELSE
   DPDEP.NEED = ERROR.DEP.NEED
END
CALL *WRITEREC('',MAT DPREC,DEPENDENTS,EEID.DEPNO,'TRANS.LOG,':PROGRAM.NAME)

Extraction: Encapsulates the dependent letter tracking update logic (sent date, letter codes, need field clearing) into a single subroutine. While currently only used by PDDL, this isolates a complex multi-step update that is error-prone if duplicated.

9. Priority Order for Decoupling

PriorityExtractionImpactReason
1*SUBMIT.LETTER.REQUESTHighUsed by all 3 programs; most duplicated code block; standardizes SmartComm interface
2*CHECK.LETTER.ELIGIBILITYHighMost complex duplicated logic; eligibility bugs affect all letter types; single source of truth for filtering rules
3*LOAD.EMPLOYER.CONTEXTHigh3-way duplication; caching logic is error-prone; simplifies all 3 programs
4*ADD.NOTICE.BILLINGMedium2-way duplication (COBRAN + HIPAA.NPP); billing errors have financial impact; increment-or-create pattern is complex
5*SAVE.PROCESS.AUDITMedium2-way duplication; standardizes audit trail naming; low risk extraction
6*UPDATE.DEP.LETTER.STATUSMediumSingle program (PDDL) but complex multi-step update; isolates dependent record mutation
7*GENERATE.FI.HIPAA.NOTICELowSingle program (HIPAA.NPP); isolates legacy PCL path; enables future deprecation
8*SUBMIT.PRINT.FTPLowSingle program (HIPAA.NPP); isolates FTP concern; simplifies HIPAA.NPP main flow
