Healthcare Benefits Correspondence Automation • Pick/MultiValue (UniData) Platform

Generated April 2025

1. Executive Summary

This report provides a comprehensive analysis of the DG Sample Code Check repository, a healthcare benefits correspondence automation system built on the Pick/MultiValue (UniData) platform. The system manages three categories of regulatory notices:

  • COBRA Notices — Initial COBRA election notices for employees and dependents losing group health coverage
  • HIPAA Privacy Notices — Notice of Privacy Practices required under HIPAA regulations
  • Disabled Dependent Letters — Six letter types for the disabled dependent validation workflow (review requests, approvals, denials, recertifications, no-response)

The system comprises 3 main programs:

ProgramPurposeLetter Count
COBRANCOBRA initial notice generation via SmartComm2
HIPAA.NPPHIPAA Notice of Privacy Practices (Self-Funded via SmartComm, Fully-Insured via PCL)1
PRINT.DISABLED.DEP.LETTERSDisabled dependent letter generation via SmartComm6

These programs interact with 10+ shared data record types including EMPLOYEES, EMPLOYERS, EROPTIONS, DEPENDENTS, and ERADJ. All three programs submit correspondence through the *CREATE.CORRESP.TRACKING subroutine using the BASIC.SMARTCOMM.DOC.1 correspondence input definition.

Key findings include 8 coupling points where functional concerns are tangled across the programs, and a proposed decoupling strategy with 8 extracted shared subroutines that would reduce code duplication and improve maintainability.

   FILENO.KEY = EEFILENO<1,EEIDX>:BILLPERIODO:SUFFIX

7. Decoupling Strategy

The following 8 extractions would convert the duplicated logic into shared, cataloged subroutines:

7.1 *CHECK.LETTER.ELIGIBILITY

Proposed Signature

SUBROUTINE *CHECK.LETTER.ELIGIBILITY(IS.ELIGIBLE, MAT EEREC, EEID, TODAY, LETTER.TYPE, OPTIONS)

Extracts from:

  • COBRAN — MAIN.LOOP (lines 196–271) + SET.FUTURE.EEIDX
  • HIPAA.NPP — FILTER.EEIDS (lines 458–690)
  • PDDL — VALIDATE.DEP.DATA + CHECK.DEP.NEED

Current duplicated pattern:

* Common pattern in all 3 programs:
IF INDEX(EENEED,'X',1) > 0 THEN CONTINUE
LOCATE TODAY IN EEEFFDATE<1,1> BY 'DR' SETTING EEIDX ELSE NULL
IF EEEFFDATE<1,EEIDX> = '' AND EEIDX > 1 THEN EEIDX -= 1
* (then program-specific status/coverage checks)

How it works: Consolidates the common EENEED check, effective date location, and status filtering into a single subroutine. The LETTER.TYPE parameter ('COBRA', 'HIPAA', 'DDL') selects which additional checks apply (e.g., coverage validation for HIPAA, DEP.NEED validation for DDL). Returns IS.ELIGIBLE flag and populates EEIDX via OPTIONS.

7.2 *LOAD.EMPLOYER.CONTEXT

Proposed Signature

SUBROUTINE *LOAD.EMPLOYER.CONTEXT(LOAD.STATUS, MAT ERREC, MAT EOREC, FILENO, LAST.FILENO, TODAY)

Extracts from:

  • COBRAN — inline in MAIN.LOOP (lines 232–260)
  • HIPAA.NPP — SET.SKIP.FILENO (lines 743–775)

Current duplicated pattern:

IF EEFILENO<1,EEIDX> <> FILENO THEN
   FILENO = EEFILENO<1,EEIDX>
   CALL *READREC('',MAT ERREC,EMPLOYERS,FILENO,'')
   IF @USER.RETURN.CODE # '' THEN ... CONTINUE
   CALL *READREC('',MAT EOREC,EROPTIONS,FILENO,'')
   IF @USER.RETURN.CODE # '' THEN ... CONTINUE
END

How it works: Wraps the cached employer+eroptions read into a single subroutine. Compares FILENO to LAST.FILENO; if different, reads both records and returns status. Callers no longer duplicate the caching logic.

SUBROUTINE *SUBMIT.LETTER.REQUEST(ERROR.MESSAGES, DOCUMENT.NAME, RECIPIENT.ID, FILENO, ADDITIONAL.DEPNO, ADDITIONAL.PURPOSE, PROGRAM.NAME)

Extracts from:

  • COBRAN — SUBMIT.CORRESP.REQUEST (lines 445–594)

  • PDDL — SUBMIT.CORRESP.REQUEST (lines 766–821)

How it works: Wraps the entire CORRESP.DATA.REC assembly and *CREATE.CORRESP.TRACKING call into a single subroutine. All three programs build the same structure with only DOCUMENT.NAME, RECIPIENT.ID, and the additional dependent fields varying.

7.4 *ADD.NOTICE.BILLING

Proposed Signature

SUBROUTINE *ADD.NOTICE.BILLING(MAT EAREC, FILENO, EEID, EEIDX, MAT EEREC, MAT EOREC, MAT ERREC, DIST.FIELD, FEE.AMOUNT)

Extracts from:

  • COBRAN — DO.BILLING (lines 598–684) with DIST.FIELD='COBRACHG'
  • HIPAA.NPP — DO.BILLING with DIST.FIELD='HIPAA'

Current duplicated pattern:

* Increment-or-create loop:
SUFFIX = ''

   FILENO.KEY = FILENO:BILLPERIODO:SUFFIX
   SUFFIX += 1
   CALL *READREC('',MAT EAREC,ERADJ,FILENO.KEY,'WAIT')

   IF EADESC[13] = 'COBRA NOTICES' THEN EXIT

How it works: Parameterizes the billing subroutine with DIST.FIELD ('COBRACHG' or 'HIPAA') and FEE.AMOUNT (from EOCOBRA.NOTICE or EOHIPAAP). The increment-or-create loop and ERADJ write logic are identical.

7.8 *UPDATE.DEP.LETTER.STATUS

Proposed Signature

SUBROUTINE *UPDATE.DEP.LETTER.STATUS(MAT DPREC, EEID.DEPNO, CUR.NEED, TODAY, ERROR.DEP.NEED, PROGRAM.NAME)

Extracts from:

  • PDDL — WRITE.DEP.REC (lines 883–906) + UPDATE.DEP.NEED (lines 859–881)

Current pattern:

* UPDATE.DEP.NEED: tracks which letters were sent today
IF DPDDL.SENT<1,1> = TODAY THEN
   * append to today's letter list
   INS CUR.NEED BEFORE TEMP.DDL.LETTERS<-1>
END ELSE
   * insert new date entry
   INS TODAY BEFORE DPDDL.SENT<1,1>
   INS CUR.NEED BEFORE DPDDL.LETTERS<1,1>
END
* WRITE.DEP.REC: clears DPDEP.NEED or preserves error codes
IF MAX.ERR.NEED = 0 THEN DPDEP.NEED = '' ELSE DPDEP.NEED = ERROR.DEP.NEED
CALL *WRITEREC('',MAT DPREC,DEPENDENTS,EEID.DEPNO,'TRANS.LOG,':PROGRAM.NAME)

How it works: Combines the letter-sent tracking update and the dependent record write-back into a single transactional subroutine. Currently PDDL-specific but extracted for consistency and future reuse if other programs need to update dependent letter status.

1*SUBMIT.LETTER.REQUESTAll 3 programsHighest duplication — identical CORRESP.DATA.REC assembly and *CREATE.CORRESP.TRACKING call in all 3 programs. Single point of change for SmartComm API evolution.
2*CHECK.LETTER.ELIGIBILITYAll 3 programsCore eligibility logic (EENEED check, effective date location, status filtering) duplicated with subtle variations. Unification prevents eligibility drift.
3*LOAD.EMPLOYER.CONTEXTCOBRAN + HIPAA.NPPEmployer + EROPTIONS cached read pattern is identical. Both programs must stay in sync on caching behavior.
4*ADD.NOTICE.BILLINGCOBRAN + HIPAA.NPPERADJ increment-or-create loop is identical except for EADISTFIELD value. Financial correctness requires a single billing implementation.
5*SAVE.PROCESS.AUDITCOBRAN + HIPAA.NPPAudit list naming and saving conventions should be consistent. Low risk extraction with moderate value.
6*UPDATE.DEP.LETTER.STATUSPDDL onlyCurrently PDDL-specific but encapsulates complex dependent record update logic. Extraction improves testability and prepares for reuse.
7*GENERATE.FI.HIPAA.NOTICEHIPAA.NPP onlyIsolates legacy PCL generation path. Enables future retirement of FI path without touching orchestrator.
8*SUBMIT.PRINT.FTPHIPAA.NPP onlyFTP logic is self-contained. Extraction is low effort and enables reuse if other programs need vendor printing.
